# 代码设计文档：转包与手机端体验审查、权限管理、JDK21 可运行化

## 一、任务背景

本次任务围绕三个问题展开：

1. 从真实用户角度检查 PC 端转包管理和手机端转包模块，明确转包逻辑，并补足 PC 端“创建转包”入口不明显的问题。
2. 补齐角色和权限管理能力，让管理员可以维护用户角色与启停状态。
3. 将后端运行环境收敛到 JDK 21，保证项目可以稳定编译和测试。

## 二、真实用户视角的问题判断

### PC 端转包管理

原页面虽然有“创建转包”Tab，但用户必须点击表格中的某一行，底部才出现转包表单。这个入口对仓库操作员不直观，容易产生“没有创建入口”的感觉。

本次改动增加了两个显性入口：

- 页面右上角新增“新建转包”按钮，点击后明确进入创建转包区域。
- 创建转包区域顶部新增“源看板号”输入框，可扫码或手输源看板号后直接选中源看板。

同时在页面顶部增加转包流程说明，降低用户理解成本：

- 目标看板号留空：拆包，系统自动创建新目标看板。
- 目标看板号填写已有同物料看板：合包，将数量汇入目标看板。

### 手机端转包模块

手机端更适合现场扫码，当前流程是“扫描源看板 -> 输入数量 -> 可选目标看板 -> 确认转包”。这个方向是对的，但仍有体验风险：

- 目标看板号的含义需要用户理解业务规则，建议后续增加“拆包/合包”模式提示。
- 手机端没有转包历史入口，现场人员完成后只能看当前结果，不利于复核。
- 手机端没有基于角色的入口控制，后续应根据角色隐藏或禁用不允许的操作。

本次没有大改手机端交互，原因是扫码体验需要真机或扫码设备验证，先保证 PC 端和后端能力闭环。

## 三、转包业务设计

转包本质是“一个已入库、未封存的源看板，将一部分数量转移到另一个看板”。

### 拆包

触发条件：

- 用户不填写目标看板号。
- 或用户填写的目标看板号不存在。

系统行为：

- 源看板数量减少。
- 创建转包出库单。
- 创建新的入库单和入库明细。
- 创建新的目标看板。
- 记录转包历史。

### 合包

触发条件：

- 用户填写的目标看板号已存在。

系统校验：

- 源看板和目标看板物料必须一致。
- 供应商必须一致。
- 目标看板必须已入库。
- 目标看板不能封存。
- 目标看板不能是源看板自身。

系统行为：

- 源看板数量减少。
- 目标看板数量增加。
- 创建转包出库单。
- 不创建新的目标入库单。
- 记录转包历史。

## 四、权限管理设计

系统已有 `SysUser.role` 字段和 JWT 角色信息，但缺少管理员维护入口。本次补齐轻量闭环：

- 后端新增 `/api/admin/users` 管理接口。
- 后端通过 Spring Security 限制 `/api/admin/**` 只有 `admin` 可访问。
- PC 前端新增“用户权限”菜单，仅 `admin` 可见。
- 管理员可新增用户、修改角色、启用或禁用账号。

角色定义：

- `admin`：系统管理员，可管理用户与权限。
- `manager`：仓库经理，可查看和维护管理类数据。
- `operator`：操作员，负责日常入库、出库、转包、扫码等任务。

## 五、JDK 设计选择

选择 JDK 21。

原因：

- 项目原 `pom.xml` 已设置 Java 21。
- 本机已有 `C:\Users\86139\.jdks\ms-21.0.8`。
- 使用 JDK 21 跑 `mvn test` 已验证通过。
- 系统默认 Java 24 会触发 Lombok 与 javac 内部 API 的兼容问题，不适合作为当前运行环境。

本次调整：

- Maven 编译插件改为 `<release>21</release>`。
- 增加 `backend/mvn-jdk21.ps1`，自动切换本机 JDK 21 后执行 Maven。

常用命令：

```powershell
cd backend
.\mvn-jdk21.ps1 test
.\mvn-jdk21.ps1 spring-boot:run
```
