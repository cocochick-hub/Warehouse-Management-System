### 高风险问题

#### 1. 流程与模式定义互相冲突

**位置**：设计文档 2.1 vs 2.2.2

**问题**：
- 2.1 固定写"创建入库单+创建目标看板"
- 2.2.2 又定义合包"不创建新入库单"
- 导致开发时不清楚实际走哪条路径

**修复**：
- 重写 2.1 流程图，明确分支判定逻辑
- 新增 2.2.1/2.2.2 分别描述拆包/合包的触发条件
- 新增 2.3 合包目标校验矩阵

**状态**：✅ 已修复

---

#### 2. 并发下可能出现超转包或负库存

**位置**：TransferServiceImpl.executeTransfer()

**问题**：只有"数量足够"校验，没有写锁策略或版本控制。多人同时操作同一源看板时可能都通过校验后扣减越界。

**修复方案**：

```java
// 方案A：悲观锁
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT k FROM InboundKanbanLabel k WHERE k.kanbanNo = :no")
Optional<InboundKanbanLabel> findByKanbanNoWithLock(@Param("no") String kanbanNo);

// 方案B：乐观锁 + 原子更新条件
@Query("UPDATE InboundKanbanLabel SET labelQty = labelQty - :qty
       WHERE kanbanNo = :no AND labelQty >= :qty")
int decreaseQty(@Param("no") String kanbanNo, @Param("qty") int qty);
// 返回受影响行数，0表示库存不足或已并发修改
```

**状态**：⚠️ 待实现（建议采用方案B乐观锁+原子更新）

---

#### 3. 合包目标看板校验条件不完整

**问题**：只写了"物料一致"，缺少关键校验（同供应商、目标状态、目标≠源等）

**修复**：新增 2.3 完整校验矩阵

| 校验项 | 规则 | 错误信息 |
|--------|------|----------|
| 目标看板存在 | 必须 | 目标看板不存在将转为拆包 |
| 物料一致性 | 必须 | 目标看板物料号与源看板不一致 |
| 目标状态=已入库 | 必须 | 目标看板状态不是已入库 |
| 目标未封存 | 必须 | 目标看板已封存 |
| 目标≠源 | 必须 | 不能对源看板自身转包 |

**状态**：✅ 已修复

---

#### 4. 单据追溯口径不闭环

**问题**：合包"不创建新入库单"，审计与对账时无法追溯本次转入动作

**修复**：
- 合包时：`targetInboundDocNo` 引用目标看板关联的原入库单号
- 同时在 `outbound_order_detail` 的 remark 中记录本次转移关系

**状态**：✅ 已修复

---

### 中风险问题

#### 5. 源看板状态机定义不清

**问题**：只写了"转包/已转包"，没有定义状态迁移规则

**修复**：新增 2.5 状态机定义

| 当前状态 | 操作后 | 说明 |
|----------|--------|------|
| `不转包` | 部分转包/已转包 | 源看板有剩余/全量转出 |
| `部分转包` | 部分转包/已转包 | 再次转包 |
| `已转包` | — | 禁止操作 |

**状态**：✅ 已修复

---

#### 6. 接口语义不够明确

**问题**：通过 targetKanbanNo 是否为空推断拆包/合包，不够显式

**修复**：在文档中明确判定逻辑，并增加错误码约定

**状态**：✅ 已修复

---

#### 7. availableQty 口径不一致

**问题**：文档写"已入库且未封存"，但页面用 availableQty

**修复**：新增 5.1 availableQty 公式定义

```
availableQty = labelQty - frozenQty - outboundQty - transferQty
```

**状态**：⚠️ 文档已定义，代码层尚未实现实时计算
