## 修复的问题

### 1. FK 约束违例（DataIntegrityViolationException）

**问题**：`OutboundOrderDetail.setOutboundOrderId(0L)` / `InboundOrderDetail.setInboundOrderId(0L)` 使用了不存在的占位ID，导致数据库 FK 约束违例。

**影响**：5个测试报 `DataIntegrityViolationException`

**修复**：
```java
// 修复前
OutboundOrder outboundOrder = createOutboundOrder(...);
outboundOrderRepository.save(outboundOrder);
OutboundOrderDetail detail = createOutboundDetail(...);
detail.setOutboundOrderId(0L); // ❌ 占位ID
outboundOrderDetailRepository.save(detail);

// 修复后
OutboundOrder outboundOrder = createOutboundOrder(...);
outboundOrder = outboundOrderRepository.save(outboundOrder); // 先保存获得真实ID
OutboundOrderDetail detail = createOutboundDetail(...);
detail.setOutboundOrderId(outboundOrder.getId()); // ✅ 使用真实FK
outboundOrderDetailRepository.save(detail);
```

**状态**：✅ 已修复

### 2. `@Modifying` 查询后 JPA 实体缓存不刷新

**问题**：`decreaseLabelQty()` 和 `increaseLabelQty()` 是 `@Modifying` 查询，直接操作数据库，绕过了 JPA 一级缓存。后续 `findByKanbanNo()` 读到的是缓存中的旧数据。

**影响**：合包测试 `shouldMergeIntoExistingTarget` 返回目标数量 30 而非更新后的 80

**修复**：
```java
// 扣减源看板后
int updated = kanbanLabelRepository.decreaseLabelQty(source.getKanbanNo(), transferQty);
if (updated == 0) {
    throw new IllegalStateException("源看板数量已变化，请重试");
}
entityManager.flush();   // 将@Modifying查询的变更刷入数据库
entityManager.clear();   // 清除一级缓存

// 重新查询获取最新数据
InboundKanbanLabel freshSource = kanbanLabelRepository.findByKanbanNo(source.getKanbanNo()).orElse(source);
```

**状态**：✅ 已修复

### 3. 合包后目标看板数量未实时更新

**问题**：合包执行 `increaseLabelQty()` 后，`TransferResultDTO` 中 `targetLabel.getLabelQty()` 仍用查询时的旧值（30），而非更新后的值（80）。

**修复**：
```java
kanbanLabelRepository.increaseLabelQty(targetKanbanNo, transferQty);
entityManager.flush();
entityManager.clear();
// 重新查询目标看板以获取最新数量
targetLabel = kanbanLabelRepository.findByKanbanNo(targetKanbanNo).orElse(targetLabel);
```

**状态**：✅ 已修复

### 5. availableQty 实时计算 ✅

**问题**：`availableQty` 直接返回 `labelQty`（硬编码），未实现设计文档中定义的实时计算公式。

**实现方案**：
```
availableQty = labelQty - transferQty - frozenQty
```

**修改的文件**：
1. **Schema** (`schema.sql` / `schema-h2.sql`)：添加 `frozen_qty` 列
2. **Entity** (`InboundKanbanLabel.java`)：添加 `frozenQty` 字段
3. **Repository** (`PackageTransferRepository.java`)：添加 `sumTransferQtyBySourceKanbanNo()` JPQL 查询
4. **Service** (`TransferServiceImpl.toKanbanDTO()`)：计算 `availableQty = labelQty - transferQty - frozenQty`
5. **Service** (`SealServiceImpl.toggleSealSingle()`)：封存时设 `frozenQty=labelQty`，解封时清零
6. **Service** (`TransferServiceImpl.executeTransfer()`)：校验时使用实时 `availableQty`

**公式说明**：
- `transferQty` = 该看板作为"源"从 `package_transfer` 汇总的已转出量
- `frozenQty` = 封存时记录冻结量（解封时清零）

**测试结果**：`Tests run: 28, Failures: 0, Errors: 0, Skipped: 0` ✅
