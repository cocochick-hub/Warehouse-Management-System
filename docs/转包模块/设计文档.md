# 转包模块设计文档

## 1. 功能需求分析

### 1.1 业务背景

转包功能用于在仓库现场将一个大包装的物料拆分为多个小包装，或将多个小包装合并。典型的业务场景包括：

- 供应商送货时整箱入库，实际生产时需要拆箱使用
- 多个小批量物料合并成一个大包装便于管理
- 物料在库区内从一个看板转移到另一个看板

### 1.2 功能需求

#### 1.2.1 核心功能

1. **查询可转包看板列表**
   - 支持分页查询
   - 支持按物料编码、供应商名称筛选
   - `availableQty = labelQty - 已冻结量 - 已出库量`（见5.1可用量公式）

2. **执行转包操作**
   - 用户选择源看板
   - 输入转包数量
   - 可选输入目标看板号（留空自动生成，或指定已有看板进行合包）
   - 系统执行转包流程

3. **转包历史查询**
   - 支持分页查询
   - 支持按源看板号、目标看板号筛选
   - 显示完整的转包信息和关联的单据号

### 1.3 用户角色

| 角色 | 权限 |
|------|------|
| 系统管理员 | 全部权限 |
| 仓库经理 | 全部权限 |
| 操作员 | 执行转包、查询 |


### 2.2 两种转包模式

#### 2.2.1 拆包模式

- **触发条件**：
  - `targetKanbanNo` 为空（自动生成）
  - `targetKanbanNo` 已指定但该看板不存在
- **场景**：大包装拆分为小包装
- **单据生成**：
  - 出库单：1张（OUT开头）
  - 入库单：1张（IN开头）
  - 目标看板：新建

#### 2.2.2 合包模式

- **触发条件**：`targetKanbanNo` 已指定且该看板存在
- **场景**：小包装合并到大包装
- **单据生成**：
  - 出库单：1张（OUT开头）
  - 入库单：**不新建**（复用目标看板关联的已有入库单）
  - 目标看板：**不新建**（更新已有看板数量）

### 2.3 合包目标校验矩阵

| 校验项 | 类型 | 规则 | 错误码 | 错误信息 |
|--------|------|------|--------|----------|
| 目标看板存在 | 必须 | `targetKanbanNo` 在系统中存在 | 400 | 目标看板不存在，将转为拆包模式 |
| 物料一致性 | 必须 | `target.materialCode == source.materialCode` | 400 | 目标看板物料号与源看板不一致，不能合包 |
| 目标状态=已入库 | 必须 | `target.labelStatus == "已入库"` | 400 | 目标看板状态不是已入库，不能合包 |
| 目标未封存 | 必须 | `target.sealed == false` | 400 | 目标看板已封存，不能合包 |
| 目标≠源 | 必须 | `target.kanbanNo != source.kanbanNo` | 400 | 不能对源看板自身进行转包 |
| 供应商一致性 | 建议 | `target.supplierCode == source.supplierCode` | 400 | 目标看板供应商与源看板不一致，建议合包前确认 |

### 2.4 并发控制策略

**问题**：多人同时操作同一源看板，可能都通过校验后超转包。

**解决方案**：事务内行锁

```java
// 查询源看板时加写锁
@Lock(LockModeType.PESSIMISTIC_WRITE)
Optional<InboundKanbanLabel> findByKanbanNoWithLock(String kanbanNo);

// 或在 TransferServiceImpl 中
InboundKanbanLabel source = kanbanLabelRepository.findByKanbanNoWithLock(request.getSourceKanbanNo())
    .orElseThrow(() -> new EntityNotFoundException("看板号不存在"));

// 原子更新条件：确保扣减后不会为负
@Query("UPDATE InboundKanbanLabel SET labelQty = labelQty - :qty WHERE kanbanNo = :no AND labelQty >= :qty")
int decreaseQty(@Param("no") String kanbanNo, @Param("qty") int qty);
```

### 2.5 源看板状态机

| 当前状态 | 操作后状态 | 说明 |
|----------|------------|------|
| `null` / `不转包` | `部分转包` 或 `已转包` | 源看板有剩余 |
| `不转包` | `已转包` | 源看板全量转出 |
| `部分转包` | `部分转包` 或 `已转包` | 再次转包 |
| `已转包` | — | 禁止操作（看板已全部转出） |

## 3. 数据库设计

### 3.1 转包记录表 (package_transfer)

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | BIGINT | PK, AUTO_INCREMENT | 主键 |
| source_kanban_no | VARCHAR(50) | NOT NULL, INDEX | 源看板号 |
| target_kanban_no | VARCHAR(50) | NOT NULL, INDEX | 目标看板号 |
| transfer_qty | INT | NOT NULL, CHECK > 0 | 转移数量 |
| source_qty_before | INT | NOT NULL, CHECK >= 0 | 转移前源看板数量 |
| source_qty_after | INT | NOT NULL, CHECK >= 0 | 转移后源看板数量 |
| material_code | VARCHAR(50) | NOT NULL | 物料编码（快照） |
| material_name | VARCHAR(100) | NOT NULL | 物料名称（快照） |
| supplier_name | VARCHAR(100) | | 供应商名称（快照） |
| operator | VARCHAR(50) | | 操作人 |
| created_at | DATETIME | NOT NULL, INDEX | 创建时间 |
| source_outbound_doc_no | VARCHAR(50) | | 源出库单号 |
| target_inbound_doc_no | VARCHAR(50) | | 目标入库单号（拆包时有值，合包时引用目标看板原有单号） |
| transfer_type | VARCHAR(20) | NOT NULL | 转包类型：拆包 / 合包 |

**索引**：
- `idx_package_transfer_source(source_kanban_no, created_at)`
- `idx_package_transfer_target(target_kanban_no, created_at)`
- `idx_package_transfer_created(created_at)`

**幂等**：业务主键 `(source_kanban_no, target_kanban_no, transfer_qty, created_at)` 理论上唯一，但时间戳无法预知，暂不建唯一约束。

### 3.2 看板标签表 (inbound_kanban_label) 约束

| 字段 | 约束 | 说明 |
|------|------|------|
| kanban_no | UNIQUE | 看板号全局唯一 |
| label_qty | CHECK >= 0 | 防止负数 |
| label_status | IN (未入库, 待入库, 已入库, 已出库) | 状态机 | 已转包 |
| sealed | BOOLEAN | 封存标记 |
| transfer_status | IN (不转包, 转包, 已转包, 转入) | 转包状态 |

### 3.3 库存表 (inventory_stock) 约束

| 字段 | 约束 | 说明 |
|------|------|------|
| on_hand_qty | CHECK >= 0 | 在线量不能为负 |
| (material_code, supplier, warehouse_area) | UNIQUE | 库存定位键 |

```
POST /api/transfer/execute
```

**请求参数**：

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| sourceKanbanNo | String | **是** | 源看板号 |
| targetKanbanNo | String | **否** | 目标看板号（不传=拆包自动生成，传入已存在=合包，传入不存在=拆包新建） |
| transferQty | Integer | **是** | 转移数量，必须 > 0 |

**分支判定逻辑**：

```
if (targetKanbanNo == null || targetKanbanNo.isBlank()) {
    // 拆包：自动生成目标看板号
    transferType = "拆包"
} else {
    target = findByKanbanNo(targetKanbanNo)
    if (target == null) {
        // 拆包：用户指定了不存在的看板号 → 创建新看板
        transferType = "拆包"
    } else {
        // 合包：目标看板存在，执行合包分支
        transferType = "合包"
        // 合包校验矩阵见 2.3
    }
}
```

**错误码约定**：

| HTTP Status | code | 场景 |
|------------|------|------|
| 200 | 200 | 成功 |
| 400 | 400 | 参数错误、校验失败（物料不一致、状态不对、库存不足） |
| 404 | 404 | 源看板不存在 |
| 409 | 409 | 并发冲突（行锁超时） |
| 500 | 500 | 系统错误（单号生成失败等） |

## 5. 可用量公式与状态定义

### 5.1 availableQty 计算公式（✅ 已实现）

```
availableQty = labelQty - transferQty - frozenQty

其中：
- labelQty      = 看板标签上的总数量（InboundKanbanLabel.labelQty）
- transferQty   = 已转出的数量（从 package_transfer 汇总该看板作为"源"的 sum(transferQty)）
- frozenQty     = 冻结量（封存时 frozenQty=labelQty，解封时 frozenQty=0）
```

**实现说明**：
- `transferQty` 通过 `PackageTransferRepository.sumTransferQtyBySourceKanbanNo()` 实时查询
- `frozenQty` 通过 `InboundKanbanLabel.frozenQty` 字段记录，封存/解封时由 `SealServiceImpl` 同步更新
- 转包数量校验 (`executeTransfer`) 使用实时 `availableQty` 而非 `labelQty`

**边界**：`availableQty >= 0`（最低为0，不会出现负数）

### 5.2 transfer_status 枚举

| 值 | 说明 | 允许操作 |
|----|------|----------|
| `不转包` | 原始状态，从未转包过 | 可转包 |
| `转包` | 部分转包，仍有剩余 | 可再次转包 |
| `已转包` | 全量转出，数量归零 | 禁止操作 |
| `转入` | 作为目标看板被转入过 | 可继续作为合包目标 |

### 6.2 页面交互说明

1. **看板列表选择**
   - 点击看板行选中
   - 不可转包的看板（状态非已入库、已封存）点击时提示警告

2. **转包执行**
   - 输入转包数量（1 ~ availableQty）
   - 输入目标看板号（可选）
     - 留空 → 拆包，自动生成看板号
     - 填入已存在看板号 → 合包
     - 填入不存在看板号 → 拆包，创建新看板
   - 点击执行转包弹出确认框
   - 成功后显示结果对话框，自动切换到历史Tab

3. **历史记录Tab**
   - 显示所有转包历史，含 transfer_type 区分拆包/合包
   - 支持按源/目标看板号筛选
   - 分页展示
