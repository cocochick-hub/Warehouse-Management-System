# admin 权限与转包入库单按钮修复 - 代码设计文档

## 任务背景

本次修复来自实际使用反馈：

1. 使用 admin 登录后，部分页面提示无权限。
2. 转包页面中与“新建入库单”相关的入口点击后没有形成有效动作。

这两个问题都属于前端流程断点：权限问题发生在路由守卫读取用户角色时；转包入库单问题发生在转包结果和入库单创建页面之间缺少明确跳转。

## 根因设计

### admin 权限问题

登录状态实际存储在 `localStorage` 的 `wms_user` 中，由 `frontend/src/utils/auth.js` 统一管理。

原路由守卫直接读取了不存在的 `wms_user_info`，导致：

- 菜单侧边栏能通过 Pinia store 看到 admin。
- 路由守卫却拿不到 role。
- 访问带 `meta.roles` 的页面时被误判为无权限。

修复设计：

- 路由守卫统一使用 `getUser()`。
- 权限判断读取 `to.matched` 中所有父子路由的 `meta.roles`，避免嵌套路由父级权限失效。

### 转包入库单按钮问题

转包拆包模式下，后端已经自动创建目标入库单，但前端结果 DTO 没有返回 `targetInboundDocNo`，因此用户无法从转包结果直接进入目标入库单。

修复设计：

- 后端 `TransferResultDTO` 增加 `targetInboundDocNo`。
- `TransferServiceImpl` 在执行转包后返回目标入库单号。
- 转包成功弹窗显示目标入库单号，并提供“查看目标入库单”按钮。
- 转包页面顶部增加“新建入库单”按钮，跳转到 `/inbound/order?create=1`。
- 入库单管理页识别 `create=1` 查询参数后自动打开创建入库单弹窗。

## 涉及文件

| 文件 | 设计作用 |
| --- | --- |
| `frontend/src/router/index.js` | 修复路由权限来源和嵌套路由权限判断 |
| `frontend/src/views/transfer/Transfer.vue` | 新增入库单跳转入口和转包结果跳转 |
| `frontend/src/views/inbound/Order.vue` | 支持通过 URL 参数自动打开创建弹窗 |
| `backend/src/main/java/com/example/wms/dto/transfer/TransferResultDTO.java` | 转包结果新增目标入库单号 |
| `backend/src/main/java/com/example/wms/service/impl/TransferServiceImpl.java` | 返回转包自动生成或复用的目标入库单号 |

## 用户流程

### admin 权限访问

1. 用户登录。
2. 登录信息写入 `wms_user`。
3. 路由守卫通过 `getUser()` 读取当前角色。
4. 若当前路由或父级路由声明了 `roles`，则检查当前用户角色是否包含在允许列表中。
5. admin 可正常进入 admin / manager 范围内的页面。

### 转包后查看入库单

1. 用户在转包管理页选择源看板。
2. 执行拆包或合包。
3. 后端返回转包结果，其中包含 `targetInboundDocNo`。
4. 前端弹窗展示目标入库单号。
5. 用户点击“查看目标入库单”，跳转到入库单管理页并按单号过滤。

### 转包页新建入库单

1. 用户在转包管理页点击“新建入库单”。
2. 前端跳转 `/inbound/order?create=1&transferStatus=转包`。
3. 入库单管理页识别 `create=1`，自动打开创建入库单弹窗。
