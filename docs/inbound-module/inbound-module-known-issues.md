## 3. 逐项说明

### KI-01：`generateDocNo()` 不是分布式级唯一方案

- 现状：当前实现为 `synchronized + 时间戳 + 3 位序号后缀 + existsByDocNo`
- 已解决的部分：已明显优于最早的“时间戳 + while 重试”版本，足够支撑单机演示
- 未彻底解决的部分：如果后续变成多实例部署，`synchronized` 只能保护单进程，不能覆盖跨实例并发
- 当前口径：**本轮按单机课堂演示验收，方案可接受**

### KI-02：`inventory_stock` 只做最小库存快照

- 当前表能回答的问题：
  - 某物料在某供应商维度下当前库存是多少
  - 最近一次入库单号和最近入库时间是什么
- 当前表不能回答的问题：
  - 不同批次如何区分
  - 货放在哪个库位
  - 每一笔库存变化流水如何追溯
- 当前口径：**本轮目标是打通入库闭环和库存联动结果展示，不等同完整库存系统**

### KI-03：编辑接口未实现

- 现状：`PUT /api/inbound/orders/{id}` 在设计文档中有提及，但本轮没有纳入实际开发
- 影响：不影响核心演示链路，因为当前演示重点是“创建单据、查看详情、手工入库、状态流转、库存联动”
- 当前口径：**本轮明确不做编辑功能，避免扩散范围**

### KI-04：手工入库页供应商下拉多一次请求

- 现状：`Manual.vue` 会额外请求一轮列表数据来拼装供应商选项
- 影响：演示数据量小时几乎无感，数据量大时会增加一次无必要请求
- 当前口径：**属于性能优化项，不是功能缺陷**

### KI-05：`normalize-wheel-es` shim 兼容

- 现状：本地 `node_modules/normalize-wheel-es` 缺少入口文件，通过 `vite.config.js` alias 到 `frontend/src/shims/normalize-wheel-es.js`
- 影响：当前构建和页面运行正常，但该 shim 更像环境兜底方案
- 当前口径：**为了保证本地环境稳定演示先保留**

### KI-06：`schema.sql` 字符集使用注意

- 现状：CLI 重建数据库时，若未指定 `--default-character-set=utf8mb4`，中文初始化数据可能触发长度或乱码问题
- 影响：只在“重建库/重新导入脚本”时出现，不影响当前已初始化环境
- 当前口径：**写入文档和演示前准备，不改业务代码**

### KI-07：缺少独立主数据模块

- 现状：物料、供应商信息目前靠入库单快照字段和现有页面条件支撑
- 影响：无法像正式系统那样做统一主数据维护、下拉字典、编码标准化
- 当前口径：**本轮是入库模块最小闭环，不扩展到主数据治理**

### KI-08：PowerShell 中文 JSON 显示乱码

- 现状：使用 `Invoke-RestMethod` 测试中文请求/响应时，控制台偶尔显示乱码
- 影响：接口本身正常，浏览器前端联调不受影响
- 当前口径：**属于终端编码观感问题，不作为缺陷处理**

### KI-09：演示问答风险

- 可能被问到的问题：
  - 为什么没有批次管理
  - 为什么没有库位管理
  - 为什么没有库存流水
  - 为什么不能编辑已创建入库单
- 推荐答法：
  - 本轮按课程阶段目标优先完成“创建 -> 入库 -> 状态流转 -> 库存联动”的最小闭环
  - 批次、库位、流水、编辑能力属于第二阶段扩展范围，当前已经在文档中做了明确留存

## 4. 优先级建议
