### 当前风险

- 命令行执行 `schema.sql` 时需要显式指定 `--default-character-set=utf8mb4`，否则中文初始化数据可能触发长度异常
- 当前仓库尚无可直接复用的物料、供应商、库存表，本轮需先采用快照字段和最小库存表方案
- 前端入库页面状态筛选仍使用数字值，需在 Card 08 前统一为中文状态值
- 前端 `npm run build` 仍存在依赖解析问题，但当前不阻塞开发模式下的需求实现
- ⚠️ **CR 新发现**：前端骨架 `Order.vue` 中 status 值为数字编码（`0`/`1`/`2`），与数据库中文值（`未入库`/`部分完成`/`已完成`）不一致 — Card 05 需确定统一格式
- ⚠️ **CR 新发现**：现有 `InboundOrder.java` 实体缺 `itemCount`/`plannedTotalQty`/`actualTotalQty`/`remark` 四个字段 — Card 05 必须首先补齐

## 2. 开发目标

在现有项目骨架基础上，完成入库模块第一阶段开发，使系统具备基础可演示能力，包括：

- 入库单新增与查询
- 入库单详情查看
- 手工入库操作
- 入库状态自动流转
- 库存数量联动更新

## 3. 当前现状评估

根据当前仓库代码，现状如下：

- 后端已存在 `InboundOrder` 实体和 `InboundOrderRepository`
- 前端已存在 `frontend/src/views/inbound/Order.vue`
- 前端已存在 `frontend/src/views/inbound/Manual.vue`
- 当前前后端均为展示骨架，未完成真实业务联动
- 数据库缺少完整的入库明细表和配套接口

结论：

本模块适合采用“补全设计 + 迭代开发”的方式推进，不建议直接在现状上零散修改。

## 8. 风险清单

| 风险 | 影响 | 应对措施 |
| :--- | :--- | :--- |
| 基础数据模块未完成 | 无法真实选择供应商和物料 | 先用模拟数据或临时下拉列表 |
| 库存模块接口未完成 | 入库后无法联动库存 | 先预留服务接口，后续补接 |
| 数据库脚本乱码 | 建表和初始化维护困难 | 优先修正脚本编码 |
| 业务规则未完全明确 | 返工风险高 | 尽早确认是否允许超收、多次入库 |
| 页面组件开发量偏大 | 影响进度 | 优先完成核心主流程，样式第二优先级 |
