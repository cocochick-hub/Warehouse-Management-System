# 入库后看板库存转包链路联通修复 - 代码核心逻辑解析

## 一、看板封存状态默认值

文件：`backend/src/main/java/com/example/wms/entity/InboundKanbanLabel.java`

```java
@Column(name = "sealed")
private Boolean sealed = false;
```

解析：

`sealed` 表示看板是否封存。转包页面只应该展示“已入库且未封存”的看板。这里给实体默认值 `false`，保证新创建的看板默认可用于后续转包。

## 二、转包可选看板查询

文件：`backend/src/main/java/com/example/wms/repository/InboundKanbanLabelRepository.java`

```java
@Query("SELECT k FROM InboundKanbanLabel k WHERE k.labelStatus = :labelStatus AND (k.sealed = false OR k.sealed IS NULL)")
Page<InboundKanbanLabel> findByLabelStatusAndSealedFalse(@Param("labelStatus") String labelStatus, Pageable pageable);
```

解析：

原来的 `findByLabelStatusAndSealedFalse` 是 Spring Data JPA 派生查询，只能匹配 `sealed=false`。旧数据中很多看板的 `sealed` 是 `NULL`，业务上也应当理解为“未封存”，所以这里改为 JPQL，明确把 `false` 和 `NULL` 都纳入查询范围。

带筛选条件的查询也使用同样逻辑：

```java
@Query("SELECT k FROM InboundKanbanLabel k WHERE k.labelStatus = :labelStatus AND (k.sealed = false OR k.sealed IS NULL) AND k.materialCode LIKE CONCAT('%', :materialCode, '%')")
Page<InboundKanbanLabel> findByLabelStatusAndSealedFalseAndMaterialCodeContaining(
        @Param("labelStatus") String labelStatus, @Param("materialCode") String materialCode, Pageable pageable);
```

这样转包页面按物料编码、供应商名称筛选时，也不会漏掉旧数据。

## 三、入库生成看板时写入 sealed=false

文件：`backend/src/main/java/com/example/wms/service/impl/InboundOrderServiceImpl.java`

```java
label.setTransferStatus(defaultIfBlank(detail.getTransferStatus(), DEFAULT_TRANSFER_STATUS));
label.setLabelStatus(LABEL_STATUS_PENDING);
label.setSealed(false);
```

解析：

这是创建待入库看板时的逻辑。看板刚生成时不应当封存，所以显式写入 `false`。

收货入库时生成已入库看板也同样设置：

```java
label.setTransferStatus(defaultIfBlank(detail.getTransferStatus(), "不转包"));
label.setLabelStatus("已入库");
label.setSealed(false);
```

这一步和用户看到库存、看板的体验直接相关。入库完成后，库存会增加，看板会变成“已入库”，且未封存，后续转包列表才能查到。

## 四、转包生成目标看板时写入 sealed=false

文件：`backend/src/main/java/com/example/wms/service/impl/TransferServiceImpl.java`

```java
target.setPackageTotal(1);
target.setLabelStatus("已入库");
target.setTransferStatus("转入");
target.setSealed(false);
```

解析：

执行转包后，系统会生成新的目标看板。这个目标看板代表转入后的包装单元，也应该是已入库、未封存状态。后续如果继续转包，目标看板也能被查询出来。

## 五、转包记录看板号长度

文件：`backend/src/main/java/com/example/wms/entity/PackageTransfer.java`

```java
@Column(name = "source_kanban_no", nullable = false, length = 100)
private String sourceKanbanNo;

@Column(name = "target_kanban_no", nullable = false, length = 100)
private String targetKanbanNo;
```

解析：

真实看板号包含日期、入库单号、物料编码、包序号等信息，长度可能超过 50 字符。这里调整为 100，避免保存转包记录时发生字段截断。

## 六、旧库结构兼容

文件：`backend/src/main/resources/schema.sql`

```sql
ALTER TABLE package_transfer
ADD COLUMN source_outbound_doc_no VARCHAR(50) DEFAULT NULL COMMENT '源出库单号';

ALTER TABLE package_transfer
ADD COLUMN target_inbound_doc_no VARCHAR(50) DEFAULT NULL COMMENT '目标入库单号';

ALTER TABLE package_transfer
ADD COLUMN transfer_type VARCHAR(20) DEFAULT NULL COMMENT '转包类型：拆包/合包';

ALTER TABLE package_transfer
MODIFY COLUMN source_kanban_no VARCHAR(100) NOT NULL COMMENT '源看板号';

ALTER TABLE package_transfer
MODIFY COLUMN target_kanban_no VARCHAR(100) NOT NULL COMMENT '目标看板号（新看板号）';
```

解析：

`CREATE TABLE IF NOT EXISTS` 只能在表不存在时创建表，不能自动补齐旧表缺失字段。这里用 `ALTER TABLE` 兼容已经存在的旧数据库，解决本地运行时转包记录无法写入的问题。

## 七、完整链路理解

从用户角度看，“创建转包”依赖的不是手动初始化看板，而是入库收货后的自动产物。

核心链路如下：

1. 入库单明细决定物料、供应商、数量和包装信息。
2. 收货后系统更新库存。
3. 收货后系统生成已入库看板。
4. 转包页面查询已入库、未封存看板。
5. 用户选择源看板和转包数量。
6. 系统扣减源看板数量。
7. 系统生成目标看板、目标入库单和转包记录。
